bp-v4.txt — BPA v4: Better Gate (High-Recall / Regret-Calibrated) + Real Memory Footprint Reduction

MISSION
BPA v3 proved we can cut effective KV reads/FLOPs, but quality regresses at L=1024 and peak KV allocation doesn’t drop because cache layout remains dense. BPA v4 fixes both:
(1) Improve the gate so we preserve quality at longer context (high recall for “needs far”),
(2) Reduce true KV memory footprint (not just reads) via a sparse/segmented cache layout.

BPA REMINDER (use this exact sentence in docs/comments when needed)
Boundary-Pressure Attention is a conditional rate controller for the model’s history channel, attempting to match memory usage to the mutual information structure of language rather than to sequence length.

================================================================================
PHASE 1 — Gate improvements: change objective from “classification” to “regret”
================================================================================
Key idea:
False negative (skip far when needed) is expensive; false positive mostly wastes compute.
Train/calibrate the gate for HIGH RECALL on “needs_far” and tune budget second.

1) Define “needs_far” label:
   - Start from boundary_pressure labels already used in v2/v3 (oracle)
   - Ensure label definition is stable across L (especially 1024)

2) Train gate with asymmetric loss (weighted BCE or focal loss):
   - Weight FN much higher than FP (e.g., w_pos 5–20x; sweep)
   - Log recall, precision, FPR, and AUC

3) Budget-calibrated thresholding (replace brittle fixed thresholds):
   Implement “quantile thresholding”:
   - Gate outputs probability p(needs_far | local_features)
   - Choose threshold per batch to hit a target enabled_rate or KV budget
   - enabled_rate target may be a function of L: enabled_rate(L)
   Provide 2 modes:
   - fixed enabled_rate (e.g., 0.30, 0.45, 0.60)
   - fixed KV_read_budget (tokens/query target)

4) Hierarchical gating (coarse→fine):
   - Coarse gate uses cheap features only:
       position bucket, local entropy/margin, maybe a tiny MLP
   - Fine gate runs only if coarse says “maybe”
   - Fine gate may use richer local features (still local-only)
   Goal: improve precision without losing recall, reduce overhead.

5) Add “local disagreement / uncertainty” features (local-only):
   Add to gate inputs:
   - local logits entropy
   - margin (top1-top2)
   - optional “stability under perturbation”:
       run a cheap second local pass with dropout or small noise (optional; keep overhead minimal)
   DO NOT use far-context scores at inference-time.

Deliverable at end of Phase 1:
A gate config that, in offline eval on validation batches, achieves:
- Recall >= 0.95 on needs_far
- With an enabled_rate budget that plausibly yields >=25% KV read savings

================================================================================
PHASE 3 — True memory footprint reduction: segmented/sparse KV cache
================================================================================
Problem: v3 only reduced reads; cache remained dense and fully allocated.

Goal: Implement a cache that stores fewer KV entries for far context when gate is off.

Minimum viable implementation (choose simplest that works):
Option A: Segmented ring buffer of far-context chunks
  - Always keep local_window densely (for correctness)
  - Far context stored as a set of retained segments/chunks (IDs + tensors)
  - When gate is off: do not store far KVs beyond local_window for that step/query
  - When gate is on: store far KVs only for selected chunks (or store compressed)
  - Attention reads only retained segments.

Option B: Layerwise “keep mask” + compacted KV
  - Maintain a compacted KV tensor for far context that grows only when enabled
  - Keep an index map from logical positions → compact positions
  - If disabled, do not append to compact far cache

Implementation requirements:
- Must produce identical outputs to the dense path when “enabled_rate=1.0” (sanity)
- Must be compatible with compile mode as much as possible (avoid Python loops in forward)
- If compile breaks, document it; still complete footprint measurement.

Measurement:
- Report peak allocated memory during eval for baseline vs BPA v4 with footprint-reducing cache
- Report stored KV entry counts (tokens stored per layer) if peak alloc is noisy
- Show reduction at L=1024

Sanity tests:
- “All-on gate” matches baseline numerically (within tolerance)
- “All-off gate” matches local-only baseline numerically
- Intermediate gating doesn’t crash; stored tokens behave as expected
