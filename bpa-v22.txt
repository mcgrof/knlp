PHASE 1 — Scale Overhead Engineering (core) (4–8h)
========================================================
Goal: get g=4-like noise behavior with g=32-like kv_ratio.

We will test 3 scale-overhead strategies. Implement each as a switchable KV quant backend.

Define groups:
- payload group_size g ∈ {4, 8} (focus)
Define scale representation and amortization:
- scale dtype ∈ {fp16, fp8/e4m3 (if available), int8-logscale}
- scale sharing window S ∈ {1, 2, 4, 8, 16} tokens (amortize across positions)

Important: for K with RoPE, scales can be shared across short windows if computed on pre-RoPE or using norm-invariant scaling. Explore both (below).

Strategy 1: Token-window scale sharing (amortize by S)
1.1) Compute per-group scale on token t0, reuse for tokens t0..t0+S-1.
1.2) Variants:
- scales computed on K_postRoPE (naive, may drift)
- scales computed on K_preRoPE (better stability) then apply after RoPE (preferred)
- scales computed on |K| norms which are RoPE-invariant (best if feasible)
1.3) Evaluate quality degradation vs S and measure kv_ratio improvement.

Strategy 2: Scale quantization / compression
2.1) Store scales in fp8 (or int8 with shared exponent) instead of fp16.
2.2) Option: store log2(scale) quantized (int8), reconstruct scale = 2^(q/α).
2.3) Measure effect on quality and bytes.

Strategy 3: Shared scales across heads or channels (light sharing)
3.1) Share scales across heads within a layer (reduce scale count by H factor).
3.2) Share scales across head_dim blocks (reduce by 2x or 4x).
3.3) This is riskier for quality; treat as ablation.

For each strategy, build a byte model:
- true kv_ratio = (int4 payload bytes + scale bytes after amortization/compression) / dense bytes
Log it.

Evaluation procedure:
- Use k fixed at 4 (best protected layers) and compare:
  A) g=32,k=4 baseline
  B) g=4,k=4 with strategy variants
  C) g=4,k=2 and k=3 (only for best strategy variants)
- Quick screen at L=8192 (3 seeds).
- Validate winners at L=16384 and L=32768 (3 seeds), r1+r2.

Success criteria (Path A win):
- PASS@3% all L with kv_ratio <= 0.30
OR
- PASS@3% all L with kv_ratio <= 0.32 but with k=3 (beat structural floor via σ4 reduction)

Failure criteria:
- If any method introduces RoPE drift causing spikes at 16K/32K, log it and move on.
