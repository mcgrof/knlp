# adam-lr-02.txt
# Scientist-mode protocol to “close the dispute” on per-layer LR scaling from Adam v (diag-Fisher proxy)
# in knlp, GPT-2 family, FineWeb-Edu, AMD W7900 (ROCm). This is designed to generate robust,
# defensible evidence, including controls that can falsify the hypothesis.
#
# You are Claude Code. Use the Ralph Wiggum loop, but behave like a careful experimentalist:
# - prioritize clean comparisons, reproducibility, and falsification controls
# - avoid “just run more” without fixing confounds
# - if something is ambiguous, resolve it by measurement, not vibes
#
# -------------------------------------------------------------------
# 0) Why we are doing this (inductive bias, stated cleanly)
# -------------------------------------------------------------------
# Observation: In adam-lr-01, per-layer LR multipliers derived from Adam’s exp_avg_sq (v)
# dramatically improved early convergence on GPT-2 124M without throughput loss.
#
# Hypothesis (optimization geometry):
#  - For NLL / cross-entropy losses, the Fisher Information Matrix (F) approximates the expected Hessian (E[H])
#    near optimum under standard conditions. Adam’s v is an exponential moving average of g^2, which is a diagonal
#    proxy for diag(F).
#  - Adam already uses per-parameter preconditioning via 1/sqrt(v), but it does not equalize curvature
#    differences across layers in a blockwise sense. A layerwise statistic of v provides a cheap proxy for
#    “how curved / sensitive this layer is,” enabling a blockwise preconditioner.
#  - Per-layer scaling thus approximates a *coarse* natural-gradient / Newton-like step where we invert
#    curvature *at the block (layer) level* instead of full-matrix inversion.
#
# What “Newtonian introspection” means here:
#  - We are not claiming full Newton’s method. We are testing whether a layerwise curvature proxy (from v)
#    can reliably reweight step sizes to match curvature, producing faster optimization without harming
#    generalization or late-stage convergence.
#
# What would convince a skeptic:
#  - The effect persists across multiple seeds, longer training budgets, and model sizes.
#  - The effect beats strong controls (random multipliers with same distribution, hand-tuned heuristics).
#  - Improvements remain when evaluation is done correctly and consistently (same tokens, same data order, same metrics).
#  - Late-stage performance and generalization are not worse, or there is a clear tradeoff curve.
#
# -------------------------------------------------------------------
# 1) Non-negotiable invariants (to avoid accidental “wins”)
# -------------------------------------------------------------------
# For each comparison set, all runs MUST share:
#  - identical code commit (except when explicitly testing implementation variants)
#  - identical dataset version/shards, preprocessing, and data order
#  - identical token budget (tokens processed), not “time budget” as primary comparator
#  - identical model init distribution + seed usage (multiple seeds)
#  - identical batch size, grad accumulation, sequence length, optimizer base LR, betas, eps, wd
#  - identical scheduler behavior (warmup, cosine/linear, min_lr)
#  - identical torch.compile setting (either all on or all off)
#  - identical eval cadence, eval sample count, and eval data split
#
# If time limits vary due to system jitter, the primary axis is TOKENS, not wall time.
#
# -------------------------------------------------------------------
# 2) Primary endpoints and metrics (what we will report)
# -------------------------------------------------------------------
# Primary outcome:
#  - Validation perplexity vs tokens (curve), including AUC over a fixed token interval.
#
# Secondary outcomes:
#  - Train loss vs tokens
#  - Final validation perplexity at fixed token budgets
#  - Convergence speed: tokens-to-reach target val ppl thresholds (e.g., 500, 400, 300, etc.)
#  - Throughput: tokens/sec distribution (mean + std), and GPU utilization if available
#  - Stability: NaNs, divergence, gradient explosion events, and frequency of clipping triggers
#
# Mechanistic introspection metrics (must log):
#  - per-layer lr multiplier time series (min/median/max across layers)
#  - per-layer update norm ||Δθ_l|| and gradient norm ||g_l|| at intervals
#  - per-layer v statistic used (stat_v_l), and s_l=(stat_v_l+eps)^(1/4)
#  - optional: correlation of lr_mult with layer depth, and with update norm
#
# Statistical treatment:
#  - at least 5 seeds for the flagship comparison at one model size
#  - report mean, std, and bootstrap 95% CI of AUC(val loss) or AUC(val ppl) across seeds
#  - do not claim “unquestionable” without CI and strong controls
#
# -------------------------------------------------------------------
# 3) Experiment design overview (phased, selective)
# -------------------------------------------------------------------
# We will not run “a month of compute” blindly. We will do three phases:
#
# Phase A (Integrity & falsification controls; cheap, days):
#  - verify the effect survives strict token-budget matching and evaluation correctness
#  - run strong controls that could explain the effect without Fisher meaning
#
# Phase B (Robustness; 1–2 weeks):
#  - longer token budgets + more seeds for the best candidate setting
#  - verify late-stage convergence and generalization
#
# Phase C (Scaling; remainder if needed):
#  - extend to GPT-2 medium and/or larger; confirm the mechanism scales
#
# Exit criteria:
#  - If controls match the improvement, the “Fisher-derived” story is likely wrong or incomplete.
#  - If improvement persists and beats controls with clean stats, we declare it robust.
#
# -------------------------------------------------------------------
# 4) Implementation checks (before any expensive runs)
# -------------------------------------------------------------------
# 4.1 Verify per-layer grouping is correct
#  - confirm param-to-layer mapping matches module naming
#  - confirm embeddings, lm_head, ln_f, and each block h.i are separated cleanly
#  - confirm tied weights are handled (embedding/lm_head tie) consistently
#
# 4.2 Scheduler correctness
#  - ensure group base_lr is preserved and schedule factor applies equally
#  - ensure lr_mult multiplies the scheduled LR, not “double scheduling”
#
# 4.3 Confirm the statistic is computed from optimizer.state exp_avg_sq
#  - confirm not accidentally using gradients from current step only
#
# 4.4 Confirm logging is consistent and complete
#  - JSONL includes step, tokens processed, lr_mult dict, stat_v dict, s_ref
#  - evaluation logs include tokens, train loss, val loss, val ppl
#
# Create a “repro manifest” per run:

#    rocM/pytorch versions, torch.compile status.
#
# -------------------------------------------------------------------
# 5) Strong controls (must run, otherwise skeptics will laugh)
# -------------------------------------------------------------------
# Control C1: Random multipliers with same distribution (placebo)
#  - Every update tick, sample lr_mult_l from the empirical distribution of Fisher-based lr_mult,
#    then randomly assign to layers (shuffle layer labels).
#  - Same clamp, same power, same warmup, same update frequency.
#  - If C1 matches Fisher-based gains, then “just adding LR heterogeneity” explains it.
#
# Control C2: Static heuristic multipliers (depth-based)
#  - e.g., linearly ramp LR from early layers to late layers, with same min/max clamp.
#  - If this matches gains, it suggests a simple heuristic is enough.
#
# Control C3: Fisher-based but frozen after early phase
#  - Compute lr_mult for first N steps (e.g., 200), then freeze for rest of run.
#  - If frozen works as well, dynamic adaptation may be unnecessary, simplifying the method.
#
# Optional control C4 (if cheap):
#  - Use mean instead of median for stat_v
#  - If median is critical, it suggests outliers matter.
#
# -------------------------------------------------------------------
# 6) Candidate settings (keep it tight, don’t explode combinatorics)
# -------------------------------------------------------------------
# Based on adam-lr-01, power=1 looked stable and didn’t saturate clamps as much.
# We will test only:
#  - Baseline: AdamW, no layer LR scaling
#  - Fisher-LR: power=1, clamp=4, warmup=200 steps (NOT 20), update_every=50 (NOT 10)
#  - Fisher-LR: power=2, clamp=2 (tighter), warmup=200, update_every=50
#  - C1: random-shuffle control matching Fisher-LR power=1 distribution
#  - C2: depth-ramp heuristic with same clamp bounds
#  - C3: Fisher-LR power=1 but freeze after 200 steps
#
# Rationale for changes vs adam-lr-01:
#  - warmup=20 and update_every=10 are aggressive and might amplify early noise.
#  - for robust science, reduce adaptivity frequency and increase stabilization.
#
# -------------------------------------------------------------------
# 7) Token budgets and seeds (what makes it “unquestionable”)
# -------------------------------------------------------------------
# Model sizes:
#  - Stage 1: GPT-2 124M (current baseline)
#  - Stage 2: GPT-2 Medium (if available in repo; else next size up that is stable)
