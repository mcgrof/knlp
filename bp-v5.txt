# bp-v5.txt — BPA v5: RA-guided “Value” Signal for Boundary-Pressure Attention (No Double Compute)

## TL;DR

We are extending Boundary-Pressure Attention (BPA) with a *usage-aligned* signal from Reciprocal Attention (RA). **RA is treated as a memory-value / retention / selection signal**, not as a second attention pass. We will:

1. **Update RA docs + visualization (atomic commit)** to clearly define “surgical” RA placement for GPT-2 (FIM-trace guided head/layer selection), and codify the chosen head set in a machine-readable form.

2. **Integrate RA into BPA** as a chunk-level value prior to improve far-context selection and memory retention, while keeping boundary_pressure as the demand-side rate controller.

3. Run **clean ablations**: random vs recency vs RA-value vs RA+recency (at fixed far budget and enabled_rate), measuring KV footprint and KV read traffic, with PPL regressions strictly bounded.

Important constraints:

* **No second attention pass.**
* **No “swap Q and K and rerun.”**
* RA must be computed as a *derived statistic* from attention weights already computed in the forward pass.
* RA collection must be limited to a small “surgical” subset of (layer, head) pairs.

## Phase 1: Implement RA “value” signal extraction for BPA (No double compute)

Goal: compute a *chunk-level RA value score* online (or offline eval) using only already-computed attention weights and only for the surgical head set S.

Definitions:

* We use attention probabilities `A[l,h,t,i]` from normal causal attention (softmax output).
* Inbound mass token-level:
  `in_mass[l,h,i] += sum_{t in batch positions} A[l,h,t,i]`
* Chunk-level:
  map each key position i to chunk id c; accumulate:
  `chunk_mass[c] += in_mass[i]`
* Aggregate across surgical head set S:
  `RA_value_chunk[c] = mean_{(l,h) in S} chunk_mass[l,h,c]`
* EMA:
  `RA_value_chunk = (1-γ)*RA_value_chunk + γ*current_value`

Tasks:

1. Add `RAValueTracker` utility (new file under `utils/` or inside `gpt2/rgsa.py` if that’s where BPA lives):

   * loads surgical head list from the JSON/YAML produced in Phase 0
   * accumulates inbound mass stats for those heads only
   * supports chunk aggregation
   * supports EMA and checkpoint save/load
2. Integrate collection into forward pass with minimal overhead:

   * Only collect for the surgical heads and only for the attention probabilities already computed.
   * Do NOT store full attention matrices if not already stored; use streamed reductions:

     * per head, accumulate `colsum` over the computed attention block (within whatever keys are attended)
3. Logging:

   * W&B: log summary stats per step:

     * RA_value_chunk entropy / concentration
     * top-k chunks by RA_value (ids + values)
     * correlation with recency baseline (optional)

Acceptance criteria:

* No measurable slowdowns beyond minimal reductions.
* No second attention call, no extra Q/K projections.
* Reproducible RA_value stats on a trained checkpoint.

## Phase 2: Use RA to improve BPA far selection and/or retention

We keep boundary_pressure as the demand-side signal (“open far or not”), and RA_value as the supply-side value signal (“which far to keep/attend”).

Two integration points (implement both if feasible; otherwise do selection first):

A) Far chunk selection when far is enabled:

* When boundary_pressure enables far for a query block, select far chunks by one of:

  1. random
  2. recency
  3. RA_value
  4. RA_value + recency blend:
     `score = RA_value * exp(-age/τ)` or `score = w1*RA + w2*recency_rank`
* Always enforce exact budget match.

B) Retention / eviction (KV footprint reduction):

* Maintain a fixed far-cache capacity (chunks stored).
* If adding a new chunk would exceed capacity, evict lowest RA_value (with optional recency floor).
* This should reduce peak KV footprint at long L.

Acceptance criteria:

* “Selection” path works end-to-end and produces deterministic chunk sets.
* “Retention” path produces measurable peak KV savings without catastrophic PPL regression.

## Phase 3: Ablations and measurements (the point of v5)

We want to measure: “Does RA-value improve quality-at-fixed-budget and/or allow more KV savings?”

Run at least these ablations at L=1024 (and optionally 512 for sanity), with 3 seeds if possible:

Fixed:

* Same dataset cap, same training/eval schedule, same enabled_rate schedule for BPA.
* Same far budget per query when far is enabled.

Variants:

1. BPA baseline (boundary_pressure gate + far selection = recency)
2. BPA + random far selection
3. BPA + RA_value far selection
4. BPA + RA_value+recency blend
5. (optional) BPA + RA_value eviction policy (retention)

Metrics to report (must be in final report):

* PPL vs tokens seen, mean±std across seeds
* Effective KV kept (tokens or chunks) as a function of L
* KV read bytes proxy (or “attended tokens/query”)
* Peak KV footprint (bytes)
* Gate overhead (ms/token)
* If time: FLOPs proxy (attended tokens/query × head_dim × layers)

Stopping rules:

* If RA_value selection does not beat recency or ties within noise at fixed budget: keep RA as an analysis tool only, do not ship it into gating.
* If RA adds overhead that wipes out memory gains: restrict RA to fewer heads or compute less frequently (e.g., every N steps).
