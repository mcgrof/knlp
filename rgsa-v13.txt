

**rgsa-v13.txt — FineWebEdu RGSA v13: Constraints-First, L²M-Guided, Random-vs-Learned Confirmations**

Context / Ground Truth So Far

* TinyStories scientific validation showed: learned routing did NOT reliably beat random routing; seed variance was high; static constraints were more stable; dynamic required narrow α tuning; dense control hurt.
* “Attention needs constraints” appears correct. “Learned routing finds semantics” appears false (at least on TinyStories at this scale).
* TinyStories may be too short/simple to demand long-range semantic retrieval. We will shift primary evaluation to FineWebEdu.
* We still care about retrieval-like behavior, but we will treat *constraints* (budgeting, smooth policies, gating to prevent pathologies) as the mechanism, and L²M as the north star for how constraints should scale with context length L.

Primary Goals for v13

1. Confirm on FineWebEdu whether “random routing ≈ learned routing” holds (important falsification check).
2. Use L²M lens to guide attention constraints: budgeted compute that scales sublinearly with L, smoother selection policies (threshold/ratio vs brittle top-k), and explicit gating to avoid pathological attention behaviors.
3. Produce clear, reproducible compute–quality tradeoff curves on FineWebEdu shard(s).
4. Avoid premature kernel work. We are validating mechanisms and stability.

Non-Goals

* No custom CUDA kernels, no FlashAttention hacking.
* No claims of wall-clock inference savings beyond algorithmic metrics (tokens/query, estimated attention FLOPs) unless a true sparse kernel exists.
* No TinyStories as primary evidence (we only use it for smoke tests/regression).

Global Experimental Settings (apply to all runs unless noted)

* CONFIG_GPT2_COMPILE=y always.
* Use fixed model size (GPT-2 ~124M) initially.
* Use consistent tokenizer and data pipeline across variants.
* Use identical token budget per run (prefer max_tokens seen; else matched iters × batch × seq_len).
* Eval schedule: eval_interval=50 (or fixed eval tokens), eval_iters constant.
* Log: val PPL, train loss; routing diagnostics; compute metrics; perf breakdown (forward/backward/opt ms) for sanity.
* Seeds: 1,2,3 (minimum); 1..5 if GPU allows for confirmation phases.

Dataset: FineWebEdu Staged Setup
Implement FineWebEdu training with a “shard/cap” mode to keep experiments bounded and comparable.

Stage A (Shard / Cap):

* cap so each run yields >= 4 eval points in a 2-hour budget
* use fixed context length T=1024 initially (later we vary L)

Stage B (Bigger Cap):

* 4× more tokens than Stage A, only after Stage A results are stable

Stage C (Longer Context):

* vary context length L (e.g., 1k, 2k, 4k if infrastructure supports) to test L²M-guided scaling behavior

Core Model Variants to Compare (FineWebEdu Stage A)
Baseline:

* vanilla GPT-2 attention

RGSA family (constraints-first framing):

1. RGSA-static: fixed chunk_size=64, fixed top_b, fixed local_window
2. RGSA-dynamic: chunk_size=32 baseline dynamic version (alpha controlled)
3. RGSA-random: same as RGSA-dynamic but chunk selection is random (seed-deterministic)

Important: We MUST include the random routing vs learned routing ablation on FineWebEdu.

* This is the critical “is the router actually doing semantics” test in a dataset where semantics might matter more.

Phase 1: FineWebEdu Stage A Scientific Confirmation (Random vs Learned)
Runs (seeds 1,2,3; 2 hours each):

* baseline
* rgsa_static (chunk=64, top_b=8, local_window=256)
* rgsa_dynamic_learned (chunk=32, top_b=8, local_window=256, alpha=0.5)
* rgsa_dynamic_random (same as learned but random chunk selection)

Outputs:

* Table: mean±std val PPL at matched checkpoints (by tokens seen)
* Curves: val PPL vs tokens seen (with error bands)
* Diagnostics: tokens/query stats, routing entropy, load balance, teacher recall if defined
* Key comparison: learned vs random performance gap (or lack thereof)

Acceptance Criteria (Phase 1):

* If learned routing does NOT beat random routing materially (within noise), we stop treating routing as semantic and proceed with constraints-first exploration.
* If learned routing beats random consistently (>= ~1 PPL absolute or clear margin across seeds), we reconsider routing improvements later, but v13 still focuses on constraints.

Phase 2: L²M-Guided Constraint Schedules (Budget as Function of Context Length L)
Implement “budget schedules” for far-context attention that scale with context length L, guided by L²M principle: capacity/budget should grow sublinearly with L.

Define budgets in terms of tokens attended per query:
Budget(L) = local_window + top_b(L) * chunk_size(L)

Add L²M-inspired schedules (configurable):

* Power-law budget scaling: budget ∝ L^β where β in {0.3, 0.5}
* Square-root schedule baseline: budget ≈ local + c * sqrt(L)
* Log schedule: budget ≈ local + c * log(L) (as a more aggressive constraint)

Operationalize by adjusting one knob at a time:
A) Fix chunk_size, vary top_b(L)
B) Fix top_b, vary chunk_size(L)
C) Hybrid: adjust both but keep budget targets consistent

Experiment Setup (FineWebEdu Stage C when L varies; otherwise simulate L by changing local_window/top_b budgets at fixed L=1024):

* For each schedule, sweep 2–3 points (low/med/high budget)
* Compare against fixed-budget RGSA (current best) and baseline

Outputs:

* Pareto plots: val PPL vs tokens/query; val PPL vs estimated attention FLOPs
* Stability metrics across seeds
* “Budget effectiveness”: marginal PPL gain per added attended token

Acceptance Criteria (Phase 2):

* We should identify at least one schedule that yields a better compute–quality frontier than fixed budgets (or show that fixed is sufficient).
* The chosen schedule should be stable across seeds (low variance, no catastrophic outliers).

Phase 3: Smoother Selection Policies (Replace Brittle top-k)
Implement alternatives to top-k chunk selection, because top-k is non-smooth and can cause instability / seed variance.

Add two selection policies:

1. Threshold gating (with cap): select all chunks with score >= τ, but cap at B_max

* τ can be fixed or annealed; optionally choose τ such that expected B ≈ target

2. Ratio / soft budget policy: select chunks probabilistically proportional to scores until reaching expected budget (or use top-k but with score temperature smoothing)

Keep everything else the same as rgsa_dynamic baseline (chunk=32, local=256).
Compare: top-k vs threshold vs ratio policy under matched expected tokens/query.

Runs:

* FineWebEdu Stage A cap, seeds 1–3, 2 hours each for the best 2–3 candidate settings

Outputs:

* PPL vs tokens seen
* tokens/query distribution p50/p95/p99 (watch tails)
* stability/variance

Acceptance Criteria (Phase 3):

* A smoother policy should reduce variance/outliers relative to top-k under similar compute.
* If it doesn’t, keep top-k and move on.

Phase 4: Pathology Guardrails (Prevent Attention Sink / Degenerate Patterns)
Since “constraints help” may be partly about preventing pathological attention behaviors, implement explicit guardrails.

Add at least one of:
A) Sink penalty: discourage excessive attention mass on earliest tokens or a designated sink region

* Implement as an auxiliary loss term computed from attention weights on a small subset of layers/steps (keep overhead low)
  B) Gate clamp: limit max attention weight per head to prevent collapse (soft constraint)
  C) Entropy regularizer on attention distribution within retrieved set (discourage single-chunk collapse)

Run on FineWebEdu Stage A cap, seeds 1–3:

* best RGSA configuration from Phases 2–3 (or baseline RGSA dynamic)
* with and without guardrail

Acceptance Criteria (Phase 4):

* Guardrails should improve stability (lower variance, fewer catastrophic seeds) without materially harming mean PPL.

Phase 5: Consolidation and Decision
At the end of v13, decide which story is true:

Case 1: Learned routing ≈ random routing on FineWebEdu too

* Conclusion: the benefit is constraint/regularization, not semantic retrieval
* Production path: drop learned router complexity; use static or stochastic structured sparsity + L²M-guided budget schedules; treat as compute/regularization method

Case 2: Learned routing beats random routing on FineWebEdu

* Conclusion: semantics matter at scale; routing can help, but needs stability improvements
* Production path: proceed with smoother selection + stability guardrails; consider better routing supervision only after stability is achieved

Implementation Notes / Logging Requirements

* Ensure all runs log: tokens_seen, val_ppl, tokens_per_query stats (mean/p50/p95/p99), estimated attention FLOPs proxy, routing entropy, load balance
* Keep compile enabled, ensure zero graph breaks
* Keep code changes minimal and reversible; prefer config knobs and clean ablations

Stopping Rules (Don’t Burn GPU Forever)

* If Phase 1 shows learned ≈ random with high confidence, deprioritize router training work for v13.
* If any variant repeatedly produces catastrophic outliers, treat it as unstable and do not carry it forward unless Phase 3 smoothing/Phase 4 guardrails fix it.
* If Phase 2 schedules fail to improve frontier, keep the simplest fixed-budget config and proceed.

End State Definition
rgsa-v13 is complete when:

* FineWebEdu confirms whether routing semantics exists (learned vs random)
* At least one L²M-guided budget schedule and one smoother selection policy have been tested and plotted
* Stability guardrail tested
* A clear next-step decision is made (constraints-only path vs semantic routing path)
