bp-v1.txt — Boundary-Pressure Attention (BPA): Rename + Risk-Reduction Plan (Scale, Length, Cheap Signal, PPL)
(Ralph Wiggum Loop Enabled)

Context / Ground Truth (Do Not Rewrite History)

* “RGSA” work (v13–v19) established:
  - Static allocation is dominated by seed variance and does not reliably help.
  - Adam exp_avg_sq / Fisher proxies measure parameter update geometry, not runtime state value.
  - Importance is query-conditional, not head-static.
  - v19 discovered a strong, stable conditional signal: boundary_pressure.
  - Threshold gating using boundary_pressure produced strong KL-selectivity and large compute savings.
* Therefore, the correct abstraction is NOT routing or static budgeting.
  It is conditional far-context activation driven by local-boundary stress.

We now rename the method:
RGSA → BPA (Boundary-Pressure Attention)

BPA Definition (Working)

BPA is attention where:
- Local window is always available.
- Far-context access is conditionally enabled per (layer, head, token) based on a boundary-pressure signal.
- The gating policy is threshold-based by default (top-k is disfavored unless proven necessary).

High-Level Objective (bp-v1)

Turn the v19 discovery into a robust, scalable method by addressing four risks in a controlled order:

R1) Does boundary_pressure remain predictive at larger scale?
R2) Does it hold under longer contexts?
R3) Can it be predicted cheaply without peeking at far-context scores?
R4) Does dynamic gating improve end-to-end PPL (not just KL alignment)?

We will NOT scale blindly. We will de-risk in phases with explicit stop/go criteria.

Non-Goals

- No new kernel work.
- No claims of wall-clock speedups.
- No reintroducing static global allocation.
- No “tune until it works” loops.
- No dependence on expensive oracle signals in the final method (only as evaluation tools).

Mandatory Controls (All Phases)

* Deterministic training configs (no AUTO).
* Exact budget matching wherever “compute parity” is required.
* Matched tokens_seen across comparisons.
* Multi-seed evaluation for any claim (minimum seeds 1,2,3).
* Same data shard list and ordering for compared runs.

Repository / Code Tasks (Rename + Plumbing)

========================================
RISK REDUCTION PLAN
========================================

We de-risk in the following order:
R3 (cheap signal) must be addressed before scaling up aggressively, because if boundary_pressure requires far-score peeking, it may be unusable.

However, we must FIRST establish that boundary_pressure is predictive beyond the exact v19 setup:
Start with R1 and R2 at modest scale/length, using the current boundary_pressure definition (even if not cheap).
Then solve R3 by building a cheap predictor that approximates boundary_pressure without far-score access.
Finally test R4 end-to-end PPL under compute-matched conditions.

----------------------------------------
PHASE 1 — R1: Scale Test (Does boundary_pressure remain predictive at larger scale?)
----------------------------------------

Goal:
Show boundary_pressure retains predictive power when model size increases.

Models:
- Small: GPT-2 124M (existing baseline)
- Medium: GPT-2 Medium (~355M) OR nearest available in repo
(If infrastructure supports, add GPT-2 Large later, but do not jump straight there.)

Setup:
- Fixed context length L=1024
- Same dataset (FineWebEdu staged/capped)
- Use the existing v19 oracle infrastructure for conditional ΔKL measurement.

Experiments:
1) Measure-only (no gating):
   - Run eval batches on trained checkpoints
   - Compute:
     * boundary_pressure distribution
     * ΔKL(l,h|x,t) oracle on a subset
     * correlation: Spearman ρ, AUC for ΔKL > τ
2) Gating-only evaluation (no retraining yet):
   - Apply threshold gating at eval-time only
   - Measure:
     * KL-selectivity ratio (enabled vs disabled ΔKL)
     * compute savings (% far-context enabled)

Acceptance (R1):
- boundary_pressure retains:
  ρ >= 0.45 OR AUC >= 0.68 on the larger model
- KL-selectivity ratio remains strong (>= 10×)
- No seed sign flips in correlation direction

Stop (R1):
- If correlation collapses on the larger model, BPA may be small-model artifact → stop scaling and diagnose.

Deliverables:
- bp_v1_phase1_scale_report.md
- correlation tables for both model sizes
- plots: boundary_pressure vs ΔKL scatter + AUC curves

----------------------------------------
PHASE 2 — R2: Long Context Test (Does it hold under longer contexts?)
----------------------------------------

Goal:
Verify boundary_pressure remains predictive as L increases.

Context lengths:
- L ∈ {1024, 2048, 4096} (only if infra supports)
If 4096 is too heavy, do {1024, 2048}.

Setup:
- Use one model size first (124M or 355M) to avoid combinatorial explosion.
- Use the same oracle measurement as in v19.

Experiments:
- For each L:
  * Measure correlation (ρ, AUC)
  * Measure how boundary_pressure distribution shifts with L
  * Measure KL-selectivity and compute savings with fixed threshold policy

Key question:
Does the threshold need to be L-dependent?
If yes:
- Fit a simple calibration: threshold(L) such that far-context enabled rate remains stable or meets a compute target.

Acceptance (R2):
- Correlation direction stable across L
- Either:
  a) same threshold works reasonably, OR
  b) threshold calibration is simple and monotonic (not brittle)

Stop (R2):
- If boundary_pressure loses predictive power at longer L, BPA is not a long-context mechanism → stop.

Deliverables:
- bp_v1_phase2_length_report.md
- plots: correlation vs L, enabled-rate vs L, KL-selectivity vs L

----------------------------------------
PHASE 3 — R3: Cheap Signal (Predict boundary_pressure without peeking at far-context scores)
----------------------------------------

Problem:
Current boundary_pressure may rely on information that effectively requires computing far-context attention scores.
If so, it’s not useful for compute savings.

Goal:
Define a cheap predictor that can be computed using:
- local_window only
- query state only
- small constant overhead

Candidate cheap predictors (implement all, then select):
C1) Local-boundary score gap:
    - Use only local attention scores near boundary (last k tokens inside local_window)
    - pressure_local = max(score at boundary band) - mean(score elsewhere local)
C2) Local-only uncertainty:
    - entropy of local-only logits (or margin between top-1 and top-2)
C3) “Attempted escape” proxy:
    - measure how concentrated attention becomes near boundary under local-only; high concentration implies missing context
C4) Residual norm spike:
    - per token norm or change in norm relative to EMA; proxy for “model is struggling”
C5) Lightweight learned gate (ONLY IF NECESSARY):
    - a tiny MLP on query vector + local stats → predicts boundary_pressure above threshold
    - trained to imitate oracle boundary_pressure labels (not end-to-end RL)

Evaluation protocol:
- Use oracle boundary_pressure as label (binary: above threshold)
- Compute AUC/precision-recall for each cheap predictor

Acceptance (R3):
- A cheap predictor achieves:
  AUC >= 0.75 for predicting boundary_pressure>thr (or ΔKL>τ directly)
- Overhead is negligible (no far KV score computation)

Stop (R3):
- If no cheap predictor works, BPA cannot deliver compute savings at runtime; it becomes only an analysis tool.

Deliverables:
- bp_v1_phase3_cheap_signal_report.md
- AUC table across predictors
- chosen cheap predictor definition and calibration method

----------------------------------------
PHASE 4 — R4: End-to-End PPL (Does dynamic gating improve PPL at matched compute?)
----------------------------------------

Goal:
Prove BPA helps the actual objective (val PPL / loss), not just KL alignment.

We will do two comparisons:

A) Compute-savings mode:
- Fix PPL target (or accept small degradation) and maximize compute savings.

B) Compute-parity mode (most important):
- Match compute (tokens/query proxy) between:
  1) uniform RGSA/BPA baseline
  2) BPA-gated (cheap predictor)
- Compare PPL across seeds.

Experiment design:
- Train from scratch or fine-tune? Choose minimal:
  - Start with short training runs for signal
  - Then longer runs only if short runs show stability

Metrics:
- mean±std PPL at matched tokens_seen
- tail risk across seeds
- compute metrics: enabled rate, tokens/query, FLOPs proxy
- stability: sign consistency across seeds

Acceptance (R4):
- BPA-gated achieves either:
  - better PPL at equal compute, OR
  - equal PPL at lower compute,
  AND does not introduce seed sign flips.

Stop (R4):
- If results flip by seed, BPA gating is unstable; revisit gating smoothing or conclude uniform remains best.

Deliverables:
- bp_v1_phase4_ppl_report.md
- W&B group links and summary tables
- final go/no-go recommendation for scaling up

========================================
Decision: Where and How to Scale Up
========================================

After phases 1–4, decide:

If R1 and R2 pass AND R3 passes AND R4 passes:
- Scale to larger models and longer contexts
- Consider integrating learned gate (small) only if it improves stability

If R3 fails:
- BPA cannot produce compute savings; keep as diagnostic tool only

If R4 fails but correlations are strong:
- BPA is useful for analysis and for selective evaluation; may not help PPL in this setup

If any phase shows seed sign flips:
- Treat as instability; reduce gating aggressiveness and add hysteresis (below), or stop.

Stability Guardrails (Implement only if needed)
- Hysteresis thresholds (thr_on > thr_off)
- EMA smoothing per head (avoid flicker)
- Minimum on-time once enabled for a token span
- Cap per-layer enabled heads to prevent collapse

### Phase 3: Cheap Signal (R3) - FAIL (commit a5c2343)

Tested cheap predictors that don't require far-context scores:

| Predictor          | Mean AUC | Spearman r |
|--------------------|----------|------------|
| local_boundary_gap | 0.50     | +0.02      |
| local_entropy      | 0.46     | -0.11      |
| escape_proxy       | 0.05     | -0.93      |

**Verdict: R3 FAILS**

All cheap predictors achieve near-random AUC (~0.5). The escape_proxy
has strong NEGATIVE correlation, but this doesn't help predict the
positive signal we need.

**Conclusion**: boundary_pressure is a DIAGNOSTIC signal, not a
PREDICTIVE one. The information needed to know whether far-context
matters IS the far-context scores themselves. Without computing
far-context attention, we cannot know if we need far-context.

This is a fundamental limit: BPA cannot deliver compute savings
because you can't know what you're missing without looking.

far-context. This is analogous to "you can't know what you don't know."
