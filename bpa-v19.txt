```text
bpa-v19.txt — “BitterKV Signal Bakeoff: Evaluate EVERYTHING (activation + curvature + stats) + allocate bits”

Mission:
We have a KV-quant mixed-precision harness that can PASS at 32K with ~0.33 kv_bytes_ratio (manual S2).
v18 showed Adam v̂-root4 is a real signal but not sharp enough to replace empirical S2.
v19 is a systematic “signal bakeoff”: define a catalog of candidate sensitivity signals (“BitterKV signals”), measure them, test correlation to oracle sensitivity, and test whether schedules derived from each signal can match S2 at long context with real packed KV storage.

Non-negotiables:
- Use the same “real model of value” as v16–v18.
- Headline contexts: L ∈ {8192, 16384, 32768}, plus L=4096 sanity.
- Seeds {0,1,2}. Decode_steps >= 256.
- Regimes: r1 batch=1, r2 microbatch=4. Try r3 microbatch=8 if feasible.
- Real KV storage required in headline runs (packed INT4/INT8, dequant on load).
- PASS: tol=3% across all seeds. Also report tol=1%.

Deliverables:
- artifacts/bitterkv/signals/*.json (all computed signals)
- artifacts/bitterkv/correlations.json (rank correlations vs oracle)
- artifacts/bitterkv/schedules/*.json (auto schedules per signal)
- results/v19/* (full eval logs)
- bpa_v19_scoreboard.json
- bpa_v19_final_report.md
- bpa_v19_branch_tree.md

========================================================
PHASE 0 — Baseline + Oracle Ground Truth (2–4h)
========================================================
0.1 Re-run baselines:
- dense
- INT8 everywhere
- S2 manual (v16 best)
Confirm reproducibility at L=8192/16384/32768.

0.2 Oracle “empirical sensitivity” (ground truth ranking):
For each layer ℓ:
- Apply INT4 only to layer ℓ KV (K and V), keep others INT8 (or dense, consistent).
- Evaluate ΔPPL at L=8192 and L=16384 on calibration prompts.
Define oracle_score[ℓ] = average ΔPPL (or ΔNLL) across those runs.
Also store oracle_rank.
Output: artifacts/bitterkv/oracle_empirical.json

(If 24-layer sweep at 2 lengths is too slow: do L=8192 only for oracle, but note it explicitly.)

========================================================
PHASE 1 — Compute ALL BitterKV Signals (core of v19) (3–6h)
========================================================
Compute per-layer signals (and optionally per-head versions, but layer-level is minimum).
Use a fixed calibration set: N=64 sequences, plus a long-context subset N=16 at L=16384.

Define the BitterKV catalog (implement each one as a function that returns score[ℓ]):

A) Activation-noise / “Fisher-on-activations” (HIGH priority)
A1) attnKL_noise_K: Inject quant-like noise into K at layer ℓ, measure ΔAttention KL
A2) attnKL_noise_V: Inject noise into V at layer ℓ, measure ΔAttention KL
A3) attnKL_noise_KV: Inject into both, measure ΔAttention KL
A4) logitKL_noise: Inject into KV, measure ΔLogit KL (output distribution)
A5) slope_attnKL: compute attnKL at two noise scales σ and 2σ; slope = (KL2-KL1)/σ
A6) fakeQuant_attnKL: do actual quant-dequant (INT4) at layer ℓ and measure ΔAttention KL

Notes:
- “Noise” must reflect quant bin width. Use per-channel scale from calibration (or scales used in INT4 quant).
- Compute KL on attention probs (softmax over attention logits), averaged over heads and a sampled set of query positions.
- To reduce cost: sample 8 query positions per sequence, and 2 layers at a time.

B) Attention statistics proxies (cheap forward-only)
B1) attn_entropy: average attention entropy per layer (lower entropy -> more sensitive)
B2) attn_maxprob: average max attention probability
B3) attn_top1_top2_margin: average margin in attention logits top1-top2
B4) late_binding_indicator: ratio of far-attention mass to total (or fraction beyond W_min)

C) Activation norms / scaling proxies
C1) norm_QK: average ||Q||*||K|| scale (logit scale proxy)
C2) norm_V: average ||V|| per layer
C3) residual_ratio: ||attn_out|| / ||residual|| per layer (how much attention matters)

D) Spectrum / effective rank proxies
D1) K_effective_rank: compute effective rank of K (SVD energy decay) on sampled tokens
D2) V_effective_rank similarly
D3) K_condnum proxy (ratio of top singular values)

E) Parameter-curvature proxies (Adam/grad) (already known imperfect, still include)
E1) adam_vhat_KV: mean v̂ over Wk,Wv
E2) adam_vhat_allattn: include Wo,Wq too
E3) gradnorm_KVparams: E[||∇θ loss||^2] over Wk,Wv on calibration
E4) gradnorm_KVacts: E[||∂loss/∂K||^2 + ||∂loss/∂V||^2] (requires backward)

F) Root/transform ablations (apply to any positive signal)
For signals with heavy tails (E1/E2, maybe A* too):
Apply monotone transforms and re-evaluate allocation outcomes:
- power p ∈ {1.0, 1/2, 1/4, 1/8}
- log1p
- winsorize at percentiles {95, 99} then power
- z-score then relu (for stability)
Store transformed variants as separate “signals”.

Implementation requirements:
- Put every signal in a registry dict: SIGNALS = {name: fn}
- Each fn returns score vector over layers, plus metadata (cost, used backward?, sampling config).
- Save every signal output to artifacts/bitterkv/signals/<name>.json

========================================================
PHASE 2 — Correlation + Predictive Power (1–2h)
========================================================
For each signal name:
- Spearman correlation vs oracle_score and vs oracle_rank
- Kendall tau optional
- Identify top-K overlap with oracle for K in {2,4,6,8}
Write artifacts/bitterkv/correlations.json

Selection:
- “Top signals” = highest Spearman and best top-6 overlap.
- Keep at least:
  - best activation-noise signal
  - best cheap proxy signal
  - best Adam-based signal (with best transform)
  - oracle itself

========================================================
PHASE 3 — Schedule Generation from Each Signal (2–4h)
========================================================
Goal: see which signal can reproduce S2 performance without oracle sweeps.

Define schedule families:
- Binary schedule INT4/INT8
- (Optional) ternary INT4/INT6/INT8 only if INT6 implemented; otherwise skip

Greedy allocator (uniform across signals):
- Start all INT4
- Sort layers by score desc (higher score -> protect more -> INT8)
- For k in 0..n_layers:
  schedule_k = top-k INT8, rest INT4
  quick eval at L=8192 on 3 seeds (or 1 seed for first pass, then 3 for finalists)
  choose minimal k that PASSes tol=3%
- Validate chosen schedule at L=16384 and L=32768 (full 3 seeds), r1+r2.
- Record kv_bytes_ratio, ppl_delta, latency.

Important:
- Use REAL packed KV for validation runs.
- For quick eval, simulated is allowed ONLY if you also rerun the selected schedule with real storage.

Controls:
- manual S2
- random schedule with same k
- oracle schedule (top-k from oracle)
- Adam-root4 from v18 (as baseline)

Output per signal:
- artifacts/bitterkv/schedules/<signal>_best.json
- results/v19/schedules/<signal>_best/* logs

========================================================
PHASE 4 — Head-Level Refinement (optional if time remains)
========================================================
If a signal wins but needs too many INT8 layers:
- extend to per-head scores within sensitive layers:
  protect only sensitive heads at INT8, others INT4
This requires KV storage to support per-head precision, which may be nontrivial.
If too hard, skip and just report as future work.

========================================================
PHASE 5 — Bandwidth-bound validation (A3 carryover) (1–2h)
========================================================
Pick top-2 schedules (best kv_bytes_ratio among PASS):
- run r3 microbatch=8 at L=16384
- profile attention kernel %
- check whether ms/token moves (likely compute-bound; still record)

========================================================
REPORTING

3) Correlation table of all signals
4) “Winner signals” by:
   - best correlation
   - best schedule quality
   - best kv_bytes_ratio among PASS
5) Answer the key question:
   - Can any signal replace oracle-derived S2?
   - If yes, which (activation-noise vs Adam vs cheap proxy)?
6) Concrete recommendation for v20:
   - If activation-noise wins: productize it as “forward Fisher” and automate bit allocation.
   - If Adam wins with a different transform: adopt it.
   - If nothing beats oracle: keep oracle calibration and search for cheaper approximations.

bpa_v19_branch_tree.md must include outcome branches:
A) Activation-noise signals win and match S2 with similar k:
   -> v20: optimize sampling to make it cheap; test transfer to 7B/8B.
B) Cheap proxy wins (entropy/margin):
   -> v20: build fast online predictor; test dynamic per prompt.
C) Adam transform wins (rare but possible):
   -> v20: integrate v̂-root^p into allocator; prove transfer across models.
D) None match S2:
   -> v20: hybrid signal (weighted sum of top 2) + learned regressor over signals.

STOP CONDITIONS
- Do not cherry-pick prompts.
- Do not compare simulated byte ratios as headline; validate with real packed KV.
- Do not hide failures; record catastrophic cases explicitly.
- If runtime is too long, reduce the number of sequences and increase sampling, but keep seeds.

Time budget tuning:
If running fast:
- increase calibration N to 128
- add L=24576
- expand transform set (more p values)
If running slow:
- reduce oracle to L=8192 only
- compute signals on N=32 but keep schedule validation on full harness.
```
