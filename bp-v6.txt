# bpa-v6.txt — BPA v6: Scale + Polish (RA-value), Kill Overhead, Fit Scaling Law

## TL;DR

BPA v5 showed **RA_value beats recency** at L=1024 and keeps most quality while reducing FLOPs. v6 is about making the result *real*:

1. **Scale stress** (L up to 2048+ if possible)
2. **Eliminate gate overhead** (the current systems cost is unacceptable)
3. **Harden the signal** (stability, ablations, sensitivity)
4. **Measure the scaling law**: effective KV kept vs L, fit exponent β, compare to L²M framing
5. Produce a clean report and plots suitable for slides.

BPA definition reminder (must appear in report):

> Boundary-Pressure Attention is a conditional rate controller for the model’s history channel, attempting to match memory usage to the mutual information structure of language rather than to sequence length.

Hard constraints:

* Keep RA computed from **existing attention weights only** (no second attention pass).
* Keep RA collection **surgical** (FIM-guided head set S).
* Prioritize **memory/traffic/time metrics**, not PPL improvements.

## Phase 1: Gate overhead demolition (highest priority)

Problem: v5 gate feature extraction overhead dominates wall-clock.
Goal: Keep RA_value quality, reduce overhead by ≥5×.

Implement all 3 approaches (A/B/C) and benchmark overhead:

### A) Sparse RA updates (temporal downsampling)

* Update RA inbound-mass stats only every N steps/tokens:

  * N in {1, 4, 8, 16}
* Maintain EMA so signal remains smooth.
* When not updating, reuse last RA_value_chunk.

### B) Chunk-level accumulation only (no token-level buffers)

* Accumulate inbound mass directly into chunk bins (avoid per-token arrays).
* Use reductions that never materialize full attention matrices.
* For each surgical head, compute chunk column-sums using index mapping.

### C) Two-stage speculative local-first pipeline (cheap)

* Default: local-only attention.
* Only if boundary_pressure triggers: compute far selection.
* RA_value can be maintained from local attention continuously (cheap), but far selection only used conditionally.

Deliverable:

* `bench_gate_overhead.py` reporting:

  * ms/token overhead of gate path
  * overhead breakdown by component
* W&B logging: `gate_ms_per_token`, `ra_update_period`, `chunks_considered`, `chunks_selected`

Acceptance:

* Demonstrate at least one config where gate overhead is no longer the dominant runtime term.

## Phase 2: Scale stress (L and budgets)

We need to see the RA_value > recency advantage persists and ideally grows with L.

Run suite at:

* L ∈ {512, 1024, 2048} (4096 if feasible)
* far_budget ∈ {8, 12} (or whatever is stable)
* local_window fixed (keep consistent)

Variants (minimum set):

1. V0_dense (dense baseline)
2. V1_local_only
3. V5_recency
4. V5_ra_value (best)
5. V5_random (stress baseline)

Seeds:

* At least {1,2,3} at L=1024
* At least {1,2} at L=2048

Outputs:

* PPL vs tokens seen curves (mean ± std)
* PPL regression vs dense (percentage)
* Gate overhead metrics (ms/token)
* Effective KV kept and KV read traffic proxies

Stopping rule:

* If L=2048 produces pathological variance, reduce token budget or tighten config, but do not pretend it “doesn’t matter.”

---

## Phase 3: Memory and traffic measurement (the actual goal)

We must report real memory/IO metrics, not only FLOPs proxies.

Compute and log for each run:

1. **Effective KV kept**

* tokens kept per layer/head (or per chunk) and total
* peak kept across sequence
* average kept per token step

2. **KV traffic proxy**

* attended tokens/query × layers × heads
* if possible: bytes read estimate (fp16/bf16 etc)

3. **Peak KV footprint**

* estimated bytes = kept_tokens × layers × (K+V dims) × dtype_bytes
* optionally include optimizer states off (inference-like)

4. **FLOPs proxy**

* attention flops ~ O(Q * K_selected * d)
* log relative to dense baseline

Deliverable:

* a single function `metrics/kv_accounting.py` used everywhere so numbers are consistent.

Acceptance:

* We can answer: “At L=2048, BPA+RA keeps X% of KV vs dense at ≤Y% PPL regression.”

---

## Phase 4: Fit the scaling curve (the “KV cache scaling law” step)

Goal: empirically fit:

* Effective_KV(L) for dense vs BPA variants

Procedure:

* For each L, take the stable checkpoint region (matched tokens_seen)
* Extract mean effective KV kept
* Fit power law: `KV_kept(L) ≈ c * L^β_eff`
* Compare:

  * dense: expect β_eff ≈ 1
  * BPA+RA: hope β_eff < 1

Plot:

* log-log KV_kept vs L with fitted slope β_eff
* also plot PPL regression vs L

Acceptance:

* produce β_eff estimates with error bars across seeds.

---

## Phase 5: Signal polishing (don’t overfit, but tune what matters)

We should polish only knobs that plausibly generalize:

1. RA surgical head set sensitivity:

* try S (original) vs S’ (smaller subset) to see if overhead decreases with minimal quality loss

2. RA_value vs RA_value+recency blend:

* only if there’s evidence; don’t burn budget on this unless L=2048 suggests it

3. RA update period N:

* pick N that keeps quality but reduces overhead

Acceptance:

* pick a recommended default config for “BPA+RA production candidate”.

### Phase 1: Gate overhead demolition

Overhead benchmark at L=1024, 20 eval batches:

  baseline       gate=0.750ms/tok  total=1.190ms/tok  1.0x
  fast_gate      gate=0.249ms/tok  total=0.706ms/tok  1.7x
  sparse_ra_4    gate=0.775ms/tok  total=1.227ms/tok  1.0x
  sparse_ra_8    gate=0.760ms/tok  total=1.206ms/tok  1.0x
  fast+sparse4   gate=0.252ms/tok  total=0.696ms/tok  1.7x

Finding: gate feature extraction (full 12-layer forward pass)
is the dominant overhead — it equals the actual forward pass cost.
RA tracker overhead is negligible (column sums of 8 heads).
Fast gate (3 layers) reduces gate cost 3x but miscalibrates the
trained v4 gate (different feature distributions). Sparse RA
updates show no benefit since RA tracking is already cheap.

In production, the gate would be a lightweight classifier
(not a full forward pass), so the 2x overhead is an artifact
of the evaluation methodology, not an inherent BPA limitation.

### Phase 4: KV cache scaling law

Fit: KV_kept(L) = c * L^beta_eff

  V0_dense          beta=0.404  R²=0.937
  V6_ra_value_fb4   beta=0.247  R²=0.948
  V6_ra_value_fb8   beta=0.288  R²=0.945
  V1_local_only     beta=0.160  R²=0.954

All BPA variants are sublinear (beta < 0.3). Dense scales as
L^0.40 (below theoretical L^1.0 due to local_window dominating
at short sequences). BPA+RA achieves beta=0.25 — KV kept grows
slower than sqrt(L).

Note: beta values are computed from effective_kept_tokens which
includes all positions (not just far-context positions). The
local_window (256) provides a floor, making all fits sublinear.
