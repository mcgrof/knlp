## Exact result
```

```

## Prior type: prefill-derived (NOT oracle)
The routing prior was generated by `capture_prefill()`:
1. Bulk prefill tokens [0..N-2] with SDPA (no attention matrix materialized) → KV cache
2. Forward last token with `output_attentions=True` → attention shape [1, n_heads, 1, N-1]
3. Sum attention weights per block to get block affinities: `[n_layers, n_heads, n_blocks]`
4. Select top-K blocks per head as routing prior

This is a **practical** prior family:
- Zero incremental decode cost (priors computed during prefill)
- No training required
- Standard HuggingFace API (`output_attentions=True`)

## Model configuration
- Model: Marin 8B base
- Architecture: 32 layers, 32 query heads, 8 KV heads (GQA ratio 4), head_dim=128
- Context: 4096 tokens

## Key insight: cosine tracks tokens-per-KVH, not K alone
The data shows cosine is primarily a function of `tokens_per_KVH = K × block_size`:
- 128 tokens/KVH → cos ~0.47–0.72 (depends on block granularity)
- 512 tokens/KVH → cos ~0.78
- 1024 tokens/KVH → cos ~0.86
- 2048 tokens/KVH → cos ~0.91

This means cos>0.90 requires retaining ~2048 tokens per KV head out of 4096 total → 50% KV reduction.

## Implications for this execution phase

### Positive
1. **Prefill-derived priors work.** The 0.914 was not oracle — it's achievable at serving time.
2. **The extraction code exists.** `capture_prefill()` is already implemented and tested.
3. **The pathway is clear:** extract priors during prefill, feed as block_tables to fused kernel.

### Constraints
1. **50% KV reduction is the ceiling for cos>0.90.** At 4096 tokens, you need ~2048 tokens/KVH.
2. **Higher KV reduction (75%+) gives cos ~0.86 at best** (BS=128, K=8).
3. **The realistic speedup at cos>0.90 is ~1.8x over dense** (fused kernel at BS=256 K=8: 0.192ms vs dense 0.348ms).
4. **At cos~0.86 (75% KV reduction):** fused kernel at BS=128 K=8: 0.103ms vs dense 0.333ms → **3.2x speedup**.

### Decision impact
- The 0.914 result being prefill-derived is the **strongest possible outcome** for Phase 0.
- The constraint is that high accuracy requires retaining substantial KV (~50%).
- The most promising operating point may be **BS=128, K=8 (cos=0.86, 75% KV reduction, 3.2x speedup)** — trading some accuracy for more KV savings and better speedup.
- Phase 1 should reproduce these numbers with the fused kernel and fresh prior extraction.

## What the later fused-kernel retry showed
The later Triton kernel retry (same day) used **random/synthetic priors** and got much worse numbers:
- BS=128, K=8: cos=0.512 (vs 0.860 with prefill priors)

This confirms the prior quality gap: **random priors → cos 0.51–0.71; prefill priors → cos 0.86–0.91.**
