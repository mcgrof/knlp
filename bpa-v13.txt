bpa-13.txt — “Unified Tiered KV: BPA Bitter0→Bitter7”
Goal: implement ALL bitter variants (0–7) under ONE evaluation harness, on AMD W7900 (ROCm), using a long-context model (Qwen2.5-0.5B, max_ctx=32768). Produce decision-grade results, not vibes.

You already learned in v12:
- Bandwidth-bound regime exists at long L on W7900.
- Entropy-only “retrieval predictor” fails.
- Beta ~1 (linear kept tokens) under matched quality.
- Random eviction at 32K collapses quality.
So v13 pivots from heuristic eviction to tiered fidelity + learned routing, while still benchmarking the simple baselines.

HARD RULES
- GPU-only headline runs. Fail fast if ROCm/HIP not present.
- IN-RANGE only for headline tables (L <= max_ctx).
- Matched-quality comparisons only (tol=1% and tol=3% vs dense at same L).
- One harness drives every method: same prompts, same seeds, same decode steps, same metrics.

DELIVERABLES (must exist)
1) eval_harness.py (single entrypoint)
2) methods/ tiering implementations: bitter0..bitter7
3) results/ directory with JSON/CSV logs for every run
4) bpa_v13_final_report.md (headline + appendix)
5) bpa_v13_branch_tree.md (outcome-driven next steps)
6) a “scoreboard” table aggregating all bitter methods across L and regimes

========================================================

0.3 Unified harness spec:
Create eval_harness.py with CLI:
python eval_harness.py \
  --model qwen05b \
  --L 1024 2048 4096 8192 16384 32768 \
  --regimes r1 r2 \
  --methods dense bitter0 bitter1 ... bitter7 \
  --tol 0.01 0.03 \
  --decode_steps 256 \
  --seeds 0 1 2 \
  --outdir results/v13_run_<timestamp>

Regimes:
- r1: batch=1
- r2: microbatch=4 (or 8 if fits; auto-detect OOM and downshift but LOG it)

Metrics logged per run:
- ppl (or NLL/token), ppl_delta% vs dense
- decode latency p50/p95 per token
- tok/s
- KV proxy bytes/token (and total bytes read/write proxy)
- peak_gpu_alloc_mb / reserved_mb
- peak_cpu_rss_mb
- gate/controller overhead %
- kept tokens count by tier (full / MLA / KVSplice / dropped)
- scaling fit inputs: kept_tokens(L), kv_bytes(L), latency(L)
- stability: seed variance, outlier detection (catastrophic ppl)

Quality gating:
- For each (L, regime, seed): run dense once to get ppl_dense.
- A method PASS if ppl <= ppl_dense*(1+tol).
- If any seed catastrophically fails (>10x ppl_dense), mark method FAIL for that (L, tol, regime) even if other seeds pass. No hiding behind averages.

========================================================
PHASE 1 — TIERING PRIMITIVES (the actual “compression plumbing”) (2–4 hours)
========================================================
Implement three “storage primitives” that every bitter method uses via a common interface:

TokenStore interface (per layer or per global cache, but consistent):
- store_full(token_ids, K, V)
- store_mla(token_ids, K, V)  # compressed representation
- store_kvsplice(token_ids, K, V)  # latent memory representation
- drop(token_ids)
- retrieve(query, needed_positions) -> returns (K,V) in appropriate form

Important: We are not trying to invent new attention kernels here. Implement an APPROXIMATION that is consistent and measurable:
- Full KV: exact.
- MLA: approximate by projecting K,V to lower-dim latent (or group-query latent) then expanding back. If you don’t have real MLA code, implement a principled low-rank projection baseline:
  - K' = K @ Pk, V' = V @ Pv with rank r << d
  - On retrieval: K_hat = K' @ Pk^T, V_hat = V' @ Pv^T
  - Train-free initially; later variants can learn Pk/Pv.
- KVSplice: implement “splicing” as a learned or fixed mixture that reduces sequence length:
  - merge neighboring tokens into segments with a segment representation (K_seg, V_seg)
  - retrieval returns segment-level KV for far past (approx)
  - Start with a deterministic splice (e.g., average or attention-weighted merge); for bitter7 you will learn it.

This is enough to test tiering prospect without writing custom kernels.

========================================================
PHASE 2 — IMPLEMENT BITTER0–BITTER3 (attention-informed heuristics) (2–4 hours)
========================================================
Shared components:
- Always keep last W_min full KV.
- Maintain token importance scores.
- Every M tokens, enforce budgets by tier routing.

bitter0: baseline tiering
Tier0: full KV for last W_min
Tier1: full KV for top-K attention heavy hitters (global score)
Tier2: MLA for remaining beyond W_min
Tier3: drop tail only if over hard cap

bitter1: decayed heavy hitters
score_i(t)=lambda*score_i(t-1)+attn_mass_received_i(t)
Evict/reroute based on decayed scores.

bitter2: layer-weighted importance
score_i += sum_l w_l * attn_mass_l(i)
Default w_l increases with depth.

bitter3: two-signal routing
score_i = attn_mass_received_i * value_proxy_i
value_proxy can be ||V||, or attention_mass * ||V||, computed cheaply.

For these methods:
- Attention mass tracking must NOT store full maps.
- Use streaming updates:
  - For each step, collect attention weights aggregated over selected layers/heads (e.g., last 4 layers, all heads).
  - Accumulate per token “received” attention (heavy hitter score).

========================================================
PHASE 3 — IMPLEMENT BITTER4 (learned routing scorer) (3–5 hours)
========================================================
bitter4 replaces heuristics with a small learned router:
- Inputs per token: decayed_score, recency_bucket, attn_var, attn_entropy (if stable), token_age, maybe ||V|| stats
- Output: tier logits {full, MLA, KVSplice, drop}

Training signal:
- Use an offline oracle labeling procedure on a small calibration set:
  - For a given budget, greedily keep tokens that minimize NLL increase (approx oracle).
  - Use dense attention accumulation as a proxy: keep tokens with highest total attention received (oracle-lite).
- Train router to match oracle tier assignments.
- Validate: router should reproduce oracle decisions with >=70% accuracy on held-out calibration prompts (not AUC; actual tier classification accuracy).

Then run router online in decode.

========================================================
PHASE 4 — IMPLEMENT BITTER5 & BITTER6 (KVSplice-first and Distance-tier stack) (3–6 hours)
========================================================
These are the “real frontier” variants. Do them carefully.

bitter5: KVSplice-first
- Tier0: small recency full KV (W_min)
- Tier1: KVSplice latent memory ALWAYS maintained for far past (global compressed memory)
- Tier2: optional small set of heavy hitters remain full KV temporarily
- Tier3: drop full KV aggressively (but latent always exists)

Key requirement:
- Latent memory must actually influence outputs. Implement it as:
  - Additional “memory tokens” prepended/inserted to attention as compressed summaries.
  - Or a low-rank adapter to K,V for far context.
  Keep it simple, consistent, measurable.

bitter6: distance-tier compression
- Partition context by distance bands:
  - near: full KV (e.g., last 1K)
  - mid: MLA (e.g., 1K–8K)
  - far: KVSplice (e.g., >8K)
- No eviction unless over hard cap.
- This is the “don’t delete, reduce fidelity by distance” map.

Make the distance bands configurable per L so they scale with context:
- e.g., near = min(1024, L/16), mid = next L/4, far = rest

========================================================
PHASE 5 — IMPLEMENT BITTER7 (end-to-end rate–distortion controller) (6–10 hours, may run overnight)
========================================================
bitter7 is the bitter lesson endpoint: stop crafting thresholds. Train a controller for:
Objective:
minimize NLL + alpha*(KV_bytes_proxy) + beta*(peak_KV_memory_proxy)

Implementation plan:
- Use the same tier router (from bitter4) but make decisions differentiable:
  - Gumbel-softmax on tier selection per token group/segment
  - Straight-through estimator
- Train ONLY the router + compression parameters (MLA projections, KVSplice merge weights), keep base model frozen initially.
- Use short training: few thousand steps on a small dataset slice, with long contexts (4K–16K).
- Evaluate after training with the same harness.

Acceptance for bitter7:
- On at least one bandwidth-bound regime (L>=8192), achieve:
  - PASS at tol=3%
  - and reduce KV_bytes_proxy >= 25% vs dense
  - and improve or not regress p50 latency.

If training is unstable:
- Fall back to training router only (fixed compression operators) and report failure mode.

========================================================
PHASE 6 — UNIFIED SCOREBOARD + SCALING FITS (2–3 hours)
========================================================
Compute and report for each method:
- PASS/FAIL matrix over (L, tol, regime)
- mean + p95 decode latency
- KV_bytes_proxy/token
- peak GPU memory
- kept tokens by tier fractions
- scaling fits:
  - kept_tokens ~ L^beta
  - kv_bytes ~ L^gamma
  - latency ~ L^delta (in bandwidth-bound regime)
Fit only on PASS points.

Also compute “Bitter lesson alignment score”:
- # of hand-tuned thresholds / knobs
- fraction of decisions learned vs heuristic
(Just report counts; no fake scalar score needed.)
