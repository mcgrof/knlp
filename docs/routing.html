knlp routing

knlp routing is a decode-time systems technique for existing transformer models. Instead of letting every head read the full relevant KV footprint, routing restricts each head to a selected subset of KV blocks. The goal is to reduce memory traffic during autoregressive decode while preserving useful output quality.

Summary

In the current practical-prior path, routing does not require retraining or model-weight changes. It requires a routing prior, per-head block selection, and a backend kernel that can attend only to the selected blocks. The present implementation combines prefill-derived priors with a fused Triton decode kernel and an H100-tuned autotuned variant.

What routing changes

Standard dense decode asks every relevant head to read from the whole available KV region. Routing changes that rule by assigning each KV head a smaller set of candidate blocks and restricting decode-time attention to that set. A useful mental model is simple: dense decode reads everything that might matter, while routed decode tries to read only the blocks the prior says are worth checking for each head.

This matters most in autoregressive decode, where repeated KV reads become the bottleneck. The broader systems motivation is explained in AR decode bottleneck.

What routing is not

  • It is not a new trained model architecture.
  • It is not speculative decoding.
  • It is not tied to cartridges, although cartridges can carry reusable priors.
  • It does not require vLLM to exist as an offline or kernel-level research line.

Routing core, cartridges, and vLLM integration are related but separate. Routing is the main feature. Cartridges are one optional prior source, and vLLM is one optional serving-path integration. Today, the richer routing-policy logic still belongs mainly in the offline/core routing path before kernel launch, not inside vLLM itself.

Architecture

The current routing stack has one required path and two optional extensions. The required path produces a routing prior, turns it into per-head block tables, and executes routed decode in a fused Triton kernel. Cartridges and vLLM are optional layers on top.

Where routing policy lives today

Today, richer routing policy and prior-selection logic belong primarily to the offline/core routing path, before kernel launch. The natural implementation point is the path that builds priors and turns them into per-head block tables, not the Triton kernel itself.

In practice that means the routing-prior extraction and prior-to-block-table selection path is the right place for future policy work such as head-specific aggressiveness, thresholding, or other selection heuristics.

Current release guidance

Treat the current routing release as a kernel-first implementation. The promoted H100 baseline comes from bounded kernel configuration and better defaults, not from a new routing-policy abstraction. Keep the current path simple: use the fused kernel, keep the bounded search, and treat the existing H100 operating points as the release baseline.

Keep future work disciplined. Exhaust bounded kernel configuration and regime-specific scheduling first. Escalate to deeper kernel-body redesign only after the current search space stops paying off. If ideas from fused quantization transfer here, apply them first to prior construction or prior-to-block-table selection rather than to the routed decode kernel directly.

Origins and lineage

The routing line did not begin as a serving optimization project. It grew out of the earlier FIM-guided and reciprocal-attention research agenda within knlp, where the broader question was how to use signals already present in the transformer pipeline as indicators of tensor importance and structure. The guiding idea was broader than routing alone: if existing traces already reveal what matters, then those traces may be reusable for several kinds of optimization rather than only for one bespoke learned mechanism.

While working through that line, attention probability traces started to look like more direct functional priors for decode-time block selection than FIM traces alone. The research thread therefore shifted from asking only how to approximate attention structurally to asking which existing pipeline signals could guide later decode-time KV access. That shift turned routing into a serious systems candidate rather than an adjacent research curiosity.

From there, two follow-on branches emerged. One branch asked whether reusable document or prefix structure could be captured in cartridge-style sidecars or routing oracles. The other asked whether routing could ever be fast enough to matter in practice. The fused Triton kernel answered the second question by removing the Python-loop overhead that had made routed decode too expensive to be operationally interesting. Once that kernel existed, the question became whether practical priors were accurate enough to justify deeper validation and serving-path work.

The fused Triton kernel

The kernel is the center of the current implementation. The routing idea is simple, but without a fused kernel it degenerates into a Python loop that launches separate attention work for each head or selected block. The Triton path removes that overhead by computing one routed decode step in a single fused kernel launch across the selected KV blocks.

Interface — routed decode consumes a routing prior, block tables, and the existing KV cache
from routing.fused_routed_attention import fused_routed_decode, select_top_k_blocks

# derive per-head selected blocks from the routing prior
block_tables, block_counts = select_top_k_blocks(routing_prior, k=8)

# run routed decode on the existing paged KV cache
out = fused_routed_decode(
    q,
    k_cache,
    v_cache,
    block_tables,
    block_counts,
)
Kernel body — iterate over selected blocks, then over BLOCK_T token tiles inside each block
for block_idx in range(max_selected_blocks):
    active = block_idx < n_blocks
    phys_block = tl.load(Block_tables + ...)

    for t_start in range(0, block_size, BLOCK_T):
        t_offsets = t_start + tl.arange(0, BLOCK_T)
        t_mask = (t_offsets < block_size) & active

        k_tile = tl.load(K_cache + ...)
        scores = tl.sum(q_vec[None, :] * k_tile, axis=1) * scale
        scores = tl.where(t_mask, scores, -inf)

        m_new = tl.maximum(m_prev, tl.max(scores, axis=0))
        correction = tl.exp(m_prev - m_new)
        p = tl.exp(scores - m_new)

        v_tile = tl.load(V_cache + ...)
        acc = acc * correction + tl.sum(p[:, None] * v_tile, axis=0)
What the kernel actually does. It walks only the selected physical blocks for each KV head, streams through each block in BLOCK_T-sized token tiles, computes attention scores against the current query vector, maintains an online softmax, and accumulates values without leaving the fused kernel path.
H100 autotuning — bounded search over tile size, warp count, and pipeline stages
def _get_autotune_configs():
    configs = []
    for block_t in [32, 64, 128, 256]:
        for nw in [4, 8, 16]:
            for ns in [2, 3, 4]:
                configs.append(
                    triton.Config({"BLOCK_T": block_t}, num_warps=nw, num_stages=ns)
                )
    return configs

@triton.autotune(
    configs=_get_autotune_configs(),
    key=["head_dim", "block_size", "max_selected_blocks"],
)

Use the same fused kernel across the current routing release. The current H100 baseline depends on a bounded search and on better launch defaults, not on a separate kernel body. Include larger BLOCK_T values in the search space. On H100, BLOCK_T=block_size is the right default in the promoted regimes because it removes an unnecessary inner token-tile loop. Treat that as the starting point for future routing work.

Optional serving path: cartridges and vLLM

Routing does not require cartridges or vLLM, but they provide one concrete way to test routing in a serving stack. Cartridges can carry reusable routing sidecars or routing oracles, and the public cartridge-routing-phase1 branch in mcgrof/vllm is the current phase-1 serving experiment.

The current vLLM patch is cartridge-guided: it loads a prior source into runtime state and uses a narrow first-token routing path. That should not be confused with routing as a whole. If the cartridge branch disappeared, the fused routed kernel and offline prior-extraction story would still exist. What would disappear is this particular vLLM activation path.

The key implementation detail is that the routing prior is not embedded inside prefix_token_ids.json. The cartridge directory gains a new sidecar file, routing_prior.pt, and the serving configuration JSON is extended so the connector knows to load it.

Concretely, the cartridge directory gains one additional sidecar file: routing_prior.pt. The meaningful configuration change is in the connector JSON.

That is the concrete integration point. The artifact itself gains a routing sidecar, and the connector JSON changes from "load a cartridge" to "load a cartridge plus a routing prior and activate routed decode." This is why the current serving-path experiment is best understood as cartridge-guided routing rather than as a fully general online routing system.

Readers looking for the actual serving-path code should start at the public cartridge-routing-phase1 branch. The most relevant files are routing_sidecar_contract.md, scripts/generate_routing_prior.py, scripts/measure_routing_phase1.py, and vllm/distributed/kv_transfer/kv_connector/v1/cartridge_connector.py.

The same backend pieces — routed-attention state, conditional attention hooks, and request-scoped prior handling — could later support other prior sources such as live prefill-derived priors, request-level policy, or non-cartridge offline artifacts. If that happens, the cleaner design is for vLLM to ingest a richer prior or policy artifact produced upstream, rather than to invent the routing policy inside the serving stack itself.

Practical priors on A100

The practical-prior sweep established that prefill-derived priors are strong enough to justify real validation work. The routing story is no longer blocked by obviously weak priors.

Current H100 release baseline

Use the current H100 release baseline as a kernel-dominant routing result. Fresh attribution on the promoted path shows the kernel still dominates end-to-end time, so keep the interpretation simple: treat routing as a kernel-first implementation line and treat the promoted P1 branch as the current baseline for downstream validation.

Promoted H100 point

At BS=128, K=8, the promoted P1 base kernel measured 0.072 ms against a dense FlashAttention-backed SDPA baseline of 0.124 ms, corresponding to a 1.7× speedup with 75% KV reduction.

Release configuration rule

Extend the search and defaults to cover larger token tiles. In practice that means allowing BLOCK_T=256 and aligning the base kernel defaults with the H100 winners. Use that configuration as the release baseline for the current routed decode path, including the repaired BS=256, K=8 regime.

The routed path loads pre-computed KV state directly into GPU cache in ~0.1s, while the dense path must compute attention for all 4136 tokens (~1.6s). The W7900 result demonstrates that knlp routing is portable across GPU vendors with zero kernel modifications. The full bounded validation ladder is proven on ROCm: kernel correctness (S1), quality preservation (S2), CartridgeConnector serving (S3), and 15× TTFT speedup via KV injection (S4). The W7900 is the first platform where CartridgeConnector serving latency has been measured end-to-end.
